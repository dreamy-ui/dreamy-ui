# Theme — Accessibility Requirements

Guide for implementing and testing accessibility of Dreamy UI `DarkTheme` / `LightTheme`.

**Not a primary widget.** These are scoping wrappers (`data-theme` + class) that switch color tokens for a subtree. They must not introduce interactive semantics or break contrast / forced-colors.

---

## 1. Purpose and semantics

| Export | Behavior |
| --- | --- |
| `DarkTheme` | `data-theme="dark"` + `dark` class on a `div` |
| `LightTheme` | `data-theme="light"` + `light` class on a `div` |

- Presentational container only — **no** `role`, no focusability, no keyboard behavior.
- Do not use Theme as a substitute for a theme **toggle** control; if users switch themes, that control needs its own APG pattern (typically [Switch](https://www.w3.org/WAI/ARIA/apg/patterns/switch/) or [Button](https://www.w3.org/WAI/ARIA/apg/patterns/button/)).

---

## 2. Contrast (required)

Scoped tokens under dark/light must keep WCAG AA contrast for content inside the wrapper:

| SC | Requirement |
| --- | --- |
| [1.4.3 Contrast (Minimum)](https://www.w3.org/WAI/WCAG22/Understanding/contrast-minimum.html) | Text ≥ 4.5:1 (normal) / 3:1 (large) against themed backgrounds |
| [1.4.11 Non-text Contrast](https://www.w3.org/WAI/WCAG22/Understanding/non-text-contrast.html) | UI boundaries, icons, focus rings ≥ 3:1 |

Nesting `DarkTheme` / `LightTheme` must not leave text/icons on insufficient contrast against the **nearest** painted background.

---

## 3. Forced colors / high contrast (required)

Theme scoping must **not** break Windows High Contrast / `forced-colors` mode:

- Prefer system colors or ensure borders/focus remain visible under [`forced-colors: active`](https://developer.mozilla.org/en-US/docs/Web/CSS/@media/forced-colors).
- Avoid relying solely on background fills that disappear in forced-colors; use transparent borders / `CanvasText` patterns where needed.
- Focus indicators must remain visible ([2.4.7](https://www.w3.org/WAI/WCAG22/Understanding/focus-visible.html), [1.4.11](https://www.w3.org/WAI/WCAG22/Understanding/non-text-contrast.html)).

Do not set `color-scheme` or theme classes in a way that hides OS forced-color overrides for interactive chrome.

---

## 4. Color is not meaning

Theme changes (light ↔ dark) must not be the **only** way meaning is conveyed for status, selection, or errors inside the subtree ([1.4.1 Use of Color](https://www.w3.org/WAI/WCAG22/Understanding/use-of-color.html)).

---

## 5. WCAG 2.2 (relevant)

| SC | Level | Why |
| --- | --- | --- |
| **1.4.1 Use of Color** | A | Theme/tint alone is not meaning |
| **1.4.3 Contrast (Minimum)** | AA | Themed text/background pairs |
| **1.4.11 Non-text Contrast** | AA | Themed controls / focus |
| **1.4.12 Text Spacing** | AA | Theme wrappers must not clip reflowed text |
| **2.4.7 Focus Visible** | AA | Focus still visible in both themes / forced-colors |

---

## 6. Checklist

- [ ] Wrapper has no interactive role or tabindex
- [ ] Text contrast AA under both light and dark scopes
- [ ] Non-text / focus contrast AA under both scopes
- [ ] Forced-colors: borders and focus remain visible
- [ ] Nested themes do not strand low-contrast content
- [ ] Theme toggle (if any) is a separate accessible control
