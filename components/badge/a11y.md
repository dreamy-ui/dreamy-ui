# Badge — Accessibility Requirements

Guide for implementing and testing accessibility of the Dreamy UI `Badge` component.

**Focus:** naming and whether the badge is informative or decorative. No dedicated APG pattern — apply [non-text / status](https://www.w3.org/WAI/WCAG22/Understanding/non-text-content.html) and name/role practices.

---

## 1. Purpose and semantics

A badge is a compact label or status indicator (count, “New”, severity). Dreamy’s `Badge` is a styled `<div>` — **semantics come from content and context**, not a built-in ARIA role.

---

## 2. Component anatomy

| Part | Element (Dreamy) | Role in a11y tree | Purpose |
| --- | --- | --- | --- |
| **Badge** | `<div>` | Generic / text container | Visual chip; children provide the name |

```
div.Badge → text | icon + text
```

## 3. Requirements

### Informative vs decorative

| Use case | Requirement |
| --- | --- |
| **Informative** (status, count, category) | Visible text (or `aria-label` if icon-only) that conveys the meaning |
| **Decorative** (purely visual accent next to already-labeled UI) | Hide from AT (`aria-hidden="true"`) **or** ensure sibling text already conveys the same info |
| **Live count** (notifications) | Prefer announcing updates via a polite live region / status message when the count changes (WCAG 4.1.3) |

### Naming

- Prefer **visible text** inside the badge as the accessible name.
- Icon-only badges: provide `aria-label` (e.g. `"3 unread"`); mark decorative icons `aria-hidden`.
- Color / scheme alone must not be the only status cue (pair with text: “Error”, “Success”).
- When badge labels a control (e.g. tab with count), include badge meaning in the control’s name or keep badge text in the accessibility tree next to the control.

### Roles

- Default: no extra role (static text).
- Do not add `role="status"` unless the badge is intentionally a live status region and updates are managed carefully.
- Counts that update frequently: consider `aria-live="polite"` on a dedicated status element, not on every decorative chip.

### Keyboard

- Static badges are not focusable.
- If the whole badge is clickable, use a real `<button>` / `<a>` (or compose Badge inside one) with keyboard support and a clear name.

## 4. WCAG 2.2 (relevant)

| SC | Level | Why |
| --- | --- | --- |
| **1.1.1 Non-text Content** | A | Icon-only badges need text alternatives |
| **1.3.3 Sensory Characteristics** | A | Don’t instruct by color/position alone |
| **1.4.1 Use of Color** | A | Status not color-only |
| **1.4.3 Contrast (Minimum)** | AA | Badge text contrast |
| **4.1.2 Name, Role, Value** | A | Meaningful name when informative |
| **4.1.3 Status Messages** | AA | Dynamic counts / status updates |

---

## 5. Checklist

- [x] Informative badges have visible or `aria-label` text
- [x] Decorative badges don’t add redundant announcements
- [x] Status meaning not by color alone
- [x] Icon-only: named; icon `aria-hidden` if decorative
- [x] Interactive badges use button/link semantics
- [x] Dynamic counts announced appropriately (live/status) when required
