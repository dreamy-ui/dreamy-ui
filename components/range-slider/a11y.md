# RangeSlider — Accessibility Requirements

Guide for implementing and testing accessibility of the Dreamy UI `RangeSlider` component.

**Primary APG reference:** [Slider (Multi-Thumb) Pattern](https://www.w3.org/WAI/ARIA/apg/patterns/slidermultithumb/)

---

## Dreamy form composition

RangeSlider is a **standalone control**. Compose it **inside `Field`**. Do **not** add Field label/hint/error chrome into RangeSlider.

| Concern | Owned by | RangeSlider must |
| --- | --- | --- |
| Visible label | `Field.Label` | Name each thumb (`aria-label` pair or labelledby) |
| Hint / error | `Field.Hint` / `Field.Error` | Wire descriptions via Field / hidden inputs |
| Required / invalid / disabled | Field flags | Reflect on thumbs + inputs |

```tsx
<Field>
  <Field.Label>Price range</Field.Label>
  <RangeSlider.Root aria-label={["Minimum price", "Maximum price"]}>
    <RangeSlider.Track><RangeSlider.FilledTrack /></RangeSlider.Track>
    <RangeSlider.Thumb index={0} />
    <RangeSlider.Thumb index={1} />
  </RangeSlider.Root>
</Field>
```

---

## 1. Purpose and semantics

A multi-thumb slider selects a **range** (low and high) within min/max.

- Two thumbs, each `role="slider"`.
- Each thumb exposes its own `aria-valuenow` / `min` / `max` / optional `valuetext`.
- Thumbs must not cross unless the product explicitly allows it; keep values ordered.

---

## 2. Component anatomy (multipart)

| Part | Element (Dreamy) | Role in a11y tree | Purpose |
| --- | --- | --- | --- |
| **Root** | `<div>` | Container | Layout / orientation |
| **Hidden inputs** | two visually hidden `<input>`s | Form values | Submit low/high; Field invalid wiring |
| **Track** / **FilledTrack** | track parts | Presentational | Visual range |
| **Thumb** (`index` 0 \| 1) | thumb parts | **Two `role="slider"` widgets** | Start and end values |
| **Mark** | optional | Presentational | Tick marks |

**Rules:**

1. **Both thumbs** are interactive sliders; each needs its own accessible name.
2. Prefer distinct names: e.g. “Minimum” / “Maximum”, not a single shared name for both.
3. Field.Label names the **group**; per-thumb `aria-label` / `aria-labelledby` still required for 4.1.2.
4. Only one thumb focused at a time; Tab cycles both thumbs in order.

---

## 3. Per-part requirements

### Thumbs

| Attribute | Requirement |
| --- | --- |
| `role="slider"` | On each thumb |
| `aria-valuenow` / `min` / `max` | Per thumb; stay in sync |
| `aria-valuetext` | Optional pair when numbers alone are unclear |
| `aria-label` / `aria-labelledby` | **Required per thumb** (API supports string tuples) |
| `aria-orientation` | Match visual orientation |
| `aria-disabled` / `aria-readonly` | From props / Field |

### Hidden inputs

- One per thumb for form submission.
- Must not create extra competing focus stops for the same values.

### Track / FilledTrack / Mark

- Presentational; not focusable.

---

## 4. Keyboard interaction

Same as [Slider](https://www.w3.org/WAI/ARIA/apg/patterns/slider/), applied **per focused thumb**:

| Key | Requirement |
| --- | --- |
| **Tab** / **Shift+Tab** | Move between thumbs and siblings |
| **Arrows** | Adjust focused thumb by step |
| **Home** / **End** | Min / max for that thumb (within range constraints) |
| **Page Up** / **Page Down** | Larger step if implemented |

Pointer drag on either thumb must have a keyboard-equivalent path (WCAG 2.5.7).

---

## 5. WCAG highlights

| SC | Why |
| --- | --- |
| **1.3.1 / 4.1.2** | Two named sliders with distinct values |
| **1.4.11** | Thumbs, track, focus ≥ 3:1 |
| **2.1.1 / 2.4.7** | Both thumbs keyboard-operable; focus visible |
| **2.5.8** | Each thumb ≥ 24×24 CSS px |
| **3.3.1 / 3.3.2** | Labels/errors via Field |

---

## 6. Checklist

- [ ] Used inside **Field** (no Field chrome inside RangeSlider)
- [ ] Both thumbs: `role="slider"`, correct `aria-value*`, **distinct names**
- [ ] Tab reaches both thumbs; arrows adjust the focused thumb
- [ ] Disabled / read-only / invalid reflected
- [ ] Focus visible; thumbs do not become unlabeled when Field label is only group-level
