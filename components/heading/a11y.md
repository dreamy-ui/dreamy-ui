# Heading — Accessibility Requirements

Guide for implementing and testing accessibility of the Dreamy UI `Heading` component.

**Anatomy:** styled heading element; defaults to `<h3>` via `dreamy.h3`. Visual size/variant is separate from semantic level.

---

## 1. Purpose and semantics

Headings create the **document outline** for navigation by AT (heading lists / shortcuts). Visual style must not replace semantic level.

---

## 2. Requirements

### Use real heading elements

- Render as `h1`–`h6` (override default `h3` with `as` / polymorphic element to match outline).
- Do **not** fake headings with bold `Text`, large `div`s, or `role="heading"` unless there is no better native option — and if `role="heading"` is used, set `aria-level`.

### Level vs appearance

- Choose level by **structure**, not by CSS size.
- Example: a visually large section title mid-page may still be `h2`; a small card title may be `h3`.
- Prefer one `h1` per page (the page title).

### Outline order

- Avoid skipping levels arbitrarily (e.g. `h2` → `h4`) when an intermediate section exists.
- Nested components (`Card.Title`, `EmptyState.Title`) should fit the parent page outline.

### Accessible name

- Heading text itself is the accessible name; keep it concise and descriptive (WCAG 2.4.6).

## 3. WCAG 2.2 (relevant)

| SC | Level | Why |
| --- | --- | --- |
| **1.3.1 Info and Relationships** | A | Heading structure must be programmatic |
| **1.4.3 Contrast (Minimum)** | AA | Heading text contrast |
| **2.4.6 Headings and Labels** | AA | Descriptive headings |
| **2.4.10 Section Headings** | AAA | Optional stronger sectioning |

---

## 4. Checklist

- [ ] Uses `h1`–`h6`, not bold text alone
- [ ] Level matches page outline (not just visual size)
- [ ] Default `h3` overridden when context needs another level
- [ ] Text is descriptive
- [ ] Contrast meets AA
