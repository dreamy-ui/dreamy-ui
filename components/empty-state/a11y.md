# Empty State — Accessibility Requirements

Guide for implementing and testing accessibility of the Dreamy UI `EmptyState` compound component.

**Anatomy:** `Root` → optional `Indicator` + `Content` → `Title` (`h3`) + `Description` (`p`). Actions (if any) are composed by the consumer inside the content.

---

## 1. Purpose and semantics

Empty state explains that a view has **no data** (or a filtered-empty result) and may offer next steps. It is primarily a **structured content** pattern, not a single ARIA widget.

---

## 2. Structure

| Part | Element | Requirement |
| --- | --- | --- |
| **Root** | `div` | Landmark only if the page structure needs it; otherwise plain container |
| **Indicator** | `div` | Icon/illustration: decorative → `aria-hidden` on the graphic; meaningful → text alternative |
| **Title** | `h3` (default) | Real heading; adjust level (`as` / element) to match page outline |
| **Description** | `p` | Supporting text; must not rely on color alone |
| **Actions** | buttons/links | Separate interactive controls with their own names, roles, and keyboard support |

### Heading outline

- Prefer a semantic heading for the empty-state title so AT users can navigate by heading.
- Choose the correct level for context (page `h1` already exists → empty-state title is often `h2`/`h3`).
- Do **not** fake the title with bold `Text` / `div` only.

### Interactive empty states

- If the card-like empty state is clickable as a whole, **prefer** a clear child control (button/link) instead of making the entire root a hit target.
- If the whole surface must be interactive, give it an explicit control role, keyboard support, visible focus, and an accessible name that includes the title text (WCAG 2.5.3).

---

## 3. WCAG 2.2 (relevant)

| SC | Level | Why |
| --- | --- | --- |
| **1.3.1 Info and Relationships** | A | Title/description relationships must be programmatic |
| **1.4.1 Use of Color** | A | Emptiness not conveyed by color alone |
| **2.4.6 Headings and Labels** | AA | Clear heading / control labels |
| **2.4.7 Focus Visible** | AA | Any actions show focus |
| **2.5.8 Target Size (Minimum)** | AA | Action targets ≥ 24×24 CSS px |
| **4.1.2 Name, Role, Value** | A | Actions expose correct name/role |

---

## 4. Checklist

- [x] Title uses a real heading element at the right level
- [x] Description is readable text, not status-by-color-only
- [x] Indicator graphic is `aria-hidden` when decorative
- [x] Actions are real buttons/links with accessible names
- [x] Whole-surface click patterns still meet keyboard + name requirements
