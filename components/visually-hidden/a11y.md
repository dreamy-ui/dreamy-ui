# Visually Hidden — Accessibility

`VisuallyHidden` and `VisuallyHiddenInput` hide content visually while keeping it available to assistive technologies (or to the form/DOM as appropriate). They are **not** APG interactive widgets; they support accessible naming and progressive enhancement.

---

## Purpose

| Component | Role |
| --- | --- |
| **`VisuallyHidden`** | Hide text visually but keep it in the accessibility tree — extra context for accessible names, skip-link targets’ supporting text, or SR-only instructions |
| **`VisuallyHiddenInput`** | Visually hidden `<input>` (default `readOnly`) for form values / progressive enhancement without a visible control |

Use when sighted UI is insufficient (icon-only controls, abbreviated labels) but AT users still need text.

---

## Authoring checklist

| Do | Don't |
| --- | --- |
| Provide SR-only names next to icon buttons | Duplicate the same text visibly and in `VisuallyHidden` |
| Prefer visible labels when space allows | Hide error messages that sighted users also need |
| Keep hidden text accurate and up to date | Use `display: none` / `hidden` when you need AT access (those remove content from the a11y tree) |
| Use `VisuallyHiddenInput` for non-visible form fields intentionally | Leave focusable hidden controls in the tab order without a clear purpose |

Example pattern:

```tsx
<button type="button">
  <Icon aria-hidden="true" />
  <VisuallyHidden>Close dialog</VisuallyHidden>
</button>
```

---

## WCAG notes

| Criterion | Relevance |
| --- | --- |
| **1.3.1** Info and Relationships | Hidden text can supply names/descriptions without changing layout |
| **2.4.6** Headings and Labels | Labels must still describe purpose |
| **4.1.2** Name, Role, Value | Common technique for accessible names on icon-only controls |
