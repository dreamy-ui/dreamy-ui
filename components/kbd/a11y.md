# Kbd — Accessibility Requirements

Guide for implementing and testing accessibility of the Dreamy UI `Kbd` component.

**Anatomy:** styled container (default `div`) wrapping key label text (e.g. `⌘`, `Ctrl`, `Enter`).

---

## 1. Purpose and semantics

`Kbd` visually represents a **keyboard key or shortcut**. It is presentational content, not an interactive control.

---

## 2. Requirements

- Prefer the native HTML `<kbd>` element when possible (`as="kbd"` / polymorphic `as`) so platforms recognize keyboard-input semantics.
- Content must be real text (or an accessible name), not an image of a key without alternative text.
- Do **not** make `Kbd` focusable or clickable unless it intentionally becomes a control (rare) — then use a real button/link instead of styling alone.
- Shortcuts described only with icons/glyphs still need understandable text for AT (e.g. visible “Ctrl” or an accessible name that expands symbols when needed).
- Sufficient text contrast for the key label (WCAG 1.4.3).

### Shortcut instructions

When documenting a shortcut, ensure the full instruction is available in text (e.g. “Press Control + S to save”), not only a row of unmarked glyphs.

---

## 3. WCAG 2.2 (relevant)

| SC | Level | Why |
| --- | --- | --- |
| **1.1.1 Non-text Content** | A | Glyph-only keys need a text alternative when not obvious |
| **1.3.1 Info and Relationships** | A | Prefer semantic `<kbd>` |
| **1.4.3 Contrast (Minimum)** | AA | Key label contrast |
| **4.1.2 Name, Role, Value** | A | Only if repurposed as a control |

---

## 4. Checklist

- [ ] Uses / can map to `<kbd>` semantics
- [ ] Key names are readable as text
- [ ] Not an accidental focus target
- [ ] Contrast meets AA for label text
