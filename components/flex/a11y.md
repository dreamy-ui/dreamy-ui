# Flex — Accessibility

`Flex` is a CSS flexbox layout primitive (`div`). It is **not** an APG interactive widget. A11y is mostly about not breaking child semantics and keeping a meaningful reading order.

---

## Semantics

- Generic container; no default ARIA role.
- Flex props (`direction`, `align`, `justify`, `wrap`, …) affect layout only — they do not convey structure to assistive tech.
- Children keep their own roles. Do not add `role` to `Flex` unless you intentionally create a landmark or group.

---

## Reading order

- Prefer DOM order that matches the visual/reading order (WCAG **1.3.2**).
- Avoid using `order`, column-reverse/row-reverse, or absolute positioning when that would make the spoken order differ from the visual order in a meaningful way.
- For bidirectional or responsive layouts, verify keyboard and screen-reader order still make sense.

---

## Authoring checklist

| Do | Don't |
| --- | --- |
| Use Flex for spacing/alignment of semantic children | Treat Flex as a list, toolbar, or tablist without the correct role |
| Keep focusable children in a logical tab order | Nest controls so labels/`htmlFor` associations break |
| Add `aria-label` only when the container itself needs a name | Use Flex as a clickable surface without role + keyboard |

---

## WCAG notes

| Criterion | Relevance |
| --- | --- |
| **1.3.1** Info and Relationships | Relationships come from children (lists, headings), not flex layout |
| **1.3.2** Meaningful Sequence | DOM order should match intended sequence |
| **2.4.3** Focus Order | Tab order should follow a sensible sequence |
