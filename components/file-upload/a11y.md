# FileUpload — Accessibility Requirements

Guide for implementing and testing accessibility of the Dreamy UI `FileUpload` component.

**Primary references:** native file input semantics; [Button Pattern](https://www.w3.org/WAI/ARIA/apg/patterns/button/) for triggers; drag-and-drop must have a non-drag alternative (WCAG 2.5.7).

---

## Dreamy form composition

FileUpload is a **standalone control**. Compose it **inside `Field`**. Do **not** bake Field label/hint/error into FileUpload.

`FileUpload.Label` is an optional **inline** label for the control surface — it is **not** a replacement for `Field.Label` in form layouts.

```tsx
<Field>
  <Field.Label>Resume</Field.Label>
  <FileUpload.Root accept=".pdf">
    <FileUpload.Dropzone>
      <FileUpload.DropzoneContent>Drop files or</FileUpload.DropzoneContent>
      <FileUpload.Trigger>Browse</FileUpload.Trigger>
    </FileUpload.Dropzone>
    <FileUpload.List clearable showSize />
  </FileUpload.Root>
  <Field.Hint>PDF only, max 5 MB.</Field.Hint>
  <Field.Error>Please upload a valid file.</Field.Error>
</Field>
```

---

## 1. Purpose and semantics

Users select one or more files via the OS file picker and/or drag-and-drop.

- The real control is a visually hidden `<input type="file">` inside Root.
- Trigger / Dropzone activate that input; they must not invent a second file widget role.
- File list items are content; delete/clear controls are buttons with accessible names.

---

## 2. Component anatomy (multipart)

| Part | Element (Dreamy) | Role in a11y tree | Purpose |
| --- | --- | --- | --- |
| **Root** | `<div>` + hidden `<input type="file">` | Input is the file control | Accept, multiple, disabled, required, name |
| **Label** | `<label>` | Optional adjacent label | Must associate with the file input if used |
| **Dropzone** | `<div>` | Interactive surface | Drop target; keyboard users use Trigger |
| **Trigger** | `<button>` | Button | Opens file picker |
| **ItemGroup** / **Item** | `<ul>` / `<li>` | List of files | Announces selected files |
| **ItemName** / **ItemSizeText** | text | Content | File identity |
| **ItemPreview** / image | decorative / `img` | Preview | `alt` = file name for images; icons `aria-hidden` |
| **ItemDeleteTrigger** | `<button>` | Button | Default `aria-label={`Remove ${file.name}`}` |
| **ClearTrigger** | `<button>` | Button | Default `aria-label="Clear all files"` |
| **FileText** | status text | Content | Selected file summary / placeholder |

---

## 3. Per-part requirements

### Hidden file input

- Native `type="file"`; respect `accept`, `multiple` (`maxFiles`), `disabled`, `required`, `capture`.
- Named via Field (`id` / `htmlFor` / `aria-labelledby`) or associated Label.
- `aria-invalid` / `aria-describedby` from Field when invalid or hinted.
- Must remain operable when Dropzone is used (Trigger or equivalent activates it).

### Trigger

- Real `<button type="button">` with a clear accessible name (visible text preferred).
- Disabled when Root is disabled.

### Dropzone

- Drag-and-drop is **enhancement only** (WCAG 2.5.7): Trigger (or equivalent) must always open the picker.
- Do not rely on drag alone; expose drop state visually **and** with text if status changes.

### ItemDeleteTrigger / ClearTrigger

- Accessible name required (defaults already set; preserve if customized).
- Decorative icons: `aria-hidden="true"`.

### ItemPreviewImage

- `alt={file.name}` (or meaningful alternative); avoid empty alt if the image is the only name cue.

---

## 4. Keyboard interaction

| Key | Requirement |
| --- | --- |
| **Tab** | Reach Trigger, delete/clear buttons, and any other focusables |
| **Enter** / **Space** | Activate focused button (open picker / remove / clear) |
| **Escape** | Closes OS dialogs per platform; no trap in the component |

Dropzone must not trap focus. Disabled Root: triggers not operable.

---

## 5. Status and errors

- Rejection / validation messages belong in **Field.Error** (or a live region if dynamic status is shown outside Field).
- Do not convey accept/reject by color alone (1.4.1).
- Announce file list changes when practical (`aria-live` on a status region if UX requires it — 4.1.3).

---

## 6. Checklist

- [ ] Used inside **Field** for form labeling / errors
- [ ] Hidden file input is the authoritative control; Trigger opens it
- [ ] Drag-and-drop has a keyboard/pointer button alternative
- [ ] Delete / clear buttons have accessible names
- [ ] Preview icons hidden from AT; images have useful `alt`
- [ ] Disabled / required / invalid wired from props + Field
- [ ] Focus visible on all buttons; target size ≥ 24×24 CSS px
