# DateRangePicker — Accessibility Requirements

Guide for implementing and testing accessibility of the Dreamy UI `DateRangePicker` component.

**Primary APG references:** [Date Picker Dialog Example](https://www.w3.org/WAI/ARIA/apg/patterns/dialog-modal/examples/datepicker-dialog/); [Dialog (Modal)](https://www.w3.org/WAI/ARIA/apg/patterns/dialog-modal/); grid keyboard model ([Grid Pattern](https://www.w3.org/WAI/ARIA/apg/patterns/grid/)).

---

## Dreamy form composition

DateRangePicker is a **standalone control**. Compose the trigger/input **inside `Field`**. Do **not** add Field label/hint/error chrome into DateRangePicker.

```tsx
<Field>
  <Field.Label>Stay dates</Field.Label>
  <DateRangePicker.Root>
    <DateRangePicker.Input />
    <DateRangePicker.PopoverContent>
      <DateRangePicker.Calendar />
      <DateRangePicker.Footer />
    </DateRangePicker.PopoverContent>
  </DateRangePicker.Root>
  <Field.Hint>Select check-in and check-out.</Field.Hint>
</Field>
```

---

## 1. Purpose and semantics

Users select a **start and end date** (range). Selection is typically sequential: first click/keyboard activation sets start, second sets end (order normalized if end &lt; start).

- Trigger/Input shows the formatted range (or placeholder) and opens the popover.
- Calendar cells communicate start, end, and in-range states.
- Presets (if used) are named buttons that set a complete range.

---

## 2. Component anatomy (multipart)

| Part | Element (Dreamy) | Role in a11y tree | Purpose |
| --- | --- | --- | --- |
| **Root** | Popover + context | Open state | Wraps trigger + content |
| **Input** / **Trigger** | read-only input or button | Opens overlay | Named by Field; shows range text |
| **PopoverContent** | popover content | Overlay | Contains calendar / presets / footer |
| **Calendar** | month grid + header | Interactive calendar | Pick start then end |
| **Nav buttons** | buttons | Buttons | Previous / next month — need `aria-label` |
| **Cell buttons** | `<button>` | Day controls | Select range endpoints |
| **RangePreset** / buttons | buttons | Buttons | Named preset ranges |
| **Footer** | Cancel / Apply | Buttons | Confirm when `showFooter` |

---

## 3. Per-part requirements

### Input / Trigger

- Accessible name from **Field.Label** (or `aria-label`).
- Accessible value / visible text must include **both** ends when selected (e.g. “Jan 1, 2026 – Jan 7, 2026”), or a clear incomplete state while selecting the second date.
- Open/closed state exposed via popover trigger semantics (`aria-expanded` when available).
- Decorative calendar icon: `aria-hidden`.

### Calendar cells

- Each day button needs an accessible name that includes the full date (not only the day number).
- Expose selected start/end (and ideally in-range) in a way AT can determine — at minimum via names/states, not color alone (1.4.1).
- Disabled for min/max out-of-range dates.
- Prefer a grid structure + keyboard model consistent with DatePicker (roving tabindex, arrows). If not yet implemented, treat full keyboard parity as a **required** accessibility target.

### Header nav

- Icon-only prev/next: **required** `aria-label` (“Previous month”, “Next month”).
- Month/year title available to AT.

### Presets / Footer

- Preset buttons: accessible name = visible label; selected state programmatically exposed when active.
- Cancel / Apply: clear names; Escape should dismiss like Cancel.

### Focus management

- On open: focus a sensible control (calendar or first day).
- On close: restore focus to trigger/input.
- No keyboard trap; Escape closes.

---

## 4. Keyboard interaction

| Key | Context | Requirement |
| --- | --- | --- |
| **Enter** / **Space** | Trigger | Open |
| **Escape** | Open | Close; focus → trigger |
| **Arrows** | Calendar | Move among days (target parity with DatePicker) |
| **Enter** / **Space** | Day | Set start or end per selection phase |
| **Tab** | Open | Reach nav, presets, footer, days as applicable |

Instruct users (via Field.Hint) when selection is two-step (start then end).

---

## 5. WCAG highlights

| SC | Why |
| --- | --- |
| **1.3.1 / 4.1.2** | Name + start/end values |
| **1.4.1** | Range not by color alone |
| **2.1.1 / 2.4.7** | Keyboard operable; focus visible |
| **2.5.8** | Day / icon targets ≥ 24×24 CSS px |
| **3.3.1 / 3.3.2** | Labels/errors/hints via Field |
| **3.2.2** | Completing a range must not surprise-navigate |

---

## 6. Checklist

- [x] Used inside **Field** for label / hint / error
- [x] Trigger shows full range text; named correctly
- [x] Start/end (and in-range) not color-only
- [x] Prev/next month buttons have `aria-label`
- [x] Day buttons have full-date accessible names
- [x] Focus restore on close; Escape works
- [x] Presets and footer buttons named
- [x] Keyboard parity with single DatePicker is the bar for grid navigation
