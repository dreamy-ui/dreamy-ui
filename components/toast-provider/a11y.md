# Toast Provider — Accessibility Requirements

Brief guide for the Dreamy UI `ToastProvider` (and internal `ToastManager`).

Toast **accessibility lives on the toast live regions**, not on the provider itself. See [`../toast/a11y.md`](../toast/a11y.md).

---

## 1. Role

`ToastProvider` supplies React context (`toast`, `updateToast`, `removeToast`, pause/resume) and mounts `ToastManager`, which portals positioned stacks of `Toast` instances.

It is infrastructure: no widget role, no focus management, no keyboard pattern of its own.

---

## 2. Provider responsibilities (a11y-adjacent)

| Concern | Expectation |
| --- | --- |
| Mount point | Render toasts in a portal so they are not clipped; keep them in the accessibility tree when visible |
| Queue / stacks | Multiple toasts still announce appropriately (avoid flooding assertive live regions) |
| Timing | Pause/resume on hover supports readable auto-dismiss (see Toast guide / WCAG 2.2.1) |
| Defaults | `defaultToastProps` must not strip live-region semantics from individual toasts |

---

## 3. Checklist

- [ ] Provider wraps the app tree that calls `useToast()`
- [ ] Visible toasts remain in the a11y tree (not `aria-hidden` at the manager root)
- [ ] Per-toast live roles/announcements follow the Toast a11y guide
- [ ] Manager containers are not focus traps and use `pointer-events` carefully so they do not block the page unintentionally
