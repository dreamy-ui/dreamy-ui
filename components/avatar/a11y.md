# Avatar — Accessibility Requirements

Guide for implementing and testing accessibility of the Dreamy UI `Avatar` component.

**Focus:** naming, decorative vs informative images (WCAG 1.1.1). No dedicated APG pattern — apply [image / non-text content](https://www.w3.org/WAI/WCAG22/Understanding/non-text-content.html) practices.

---

## 1. Purpose and semantics

An avatar represents a person or entity visually (photo, initials, or fallback icon). Assistive tech must get a **name when the avatar is informative**, and must **ignore it when purely decorative**.

---

## 2. Component anatomy

| Part | Element (Dreamy) | Role in a11y tree | Purpose |
| --- | --- | --- | --- |
| **Root** | `<span>` | Container | Wraps image / fallback |
| **Image** | `<img>` | Image | Photo; `alt` from `name` when loaded |
| **Name fallback** | `<div role="img" aria-label={name}>` | Image | Initials when no photo |
| **Icon fallback** | icon with `role="img"` + `aria-label` | Image | Generic person icon |
| **Children** | e.g. status badge | Depends | Optional overlay |
| **AvatarGroup** | `Flex` `role="group"` | Group | Stacked avatars + excess count |

```
span → img[alt=name] | div[role=img][aria-label=name] | icon[role=img]
AvatarGroup[role=group] → excess text + Avatar*
```

---

## 3. Requirements

### Informative vs decorative

| Use case | Requirement |
| --- | --- |
| **Informative** (identifies a user in a list, profile, comment) | Accessible name from `name` → `alt` on image, or `aria-label` on initials/`role="img"` fallback |
| **Decorative** (photo also labeled by adjacent visible name) | Prefer `alt=""` on image **or** hide avatar from AT (`aria-hidden="true"`) so the name is not announced twice |
| **Interactive** (avatar is a button/link) | Put name on the control (`aria-label` / visible text); image inside may be decorative |

### Naming rules

- Prefer the person’s **`name`** prop as the accessible name (WCAG 2.5.3: include visible name text when a label is shown).
- Loaded image: meaningful `alt` (use `name`; avoid “Image of…” / “Avatar of…” prefixes).
- Initials fallback: `role="img"` + `aria-label={name}` (not initials alone unless that is the only identity).
- Generic icon fallback: `aria-label` / `iconLabel` describing the person or “User” — not an empty unlabeled image.
- Status badges (online, etc.): do not rely on color alone; add text / `aria-label` if status matters.

### AvatarGroup

- Group may use `role="group"` with an optional group label (`aria-label`).
- Excess indicator (`+N`) should be readable text so AT hears remaining count.
- Each child avatar still follows informative/decorative rules.

### Keyboard

- Non-interactive avatars are not focusable.
- If wrapped in a link/button, that control is the tab stop with a clear name.

## 4. WCAG 2.2 (relevant)

| SC | Level | Why |
| --- | --- | --- |
| **1.1.1 Non-text Content** | A | `alt` / name for informative avatars |
| **1.4.1 Use of Color** | A | Status overlays |
| **2.5.3 Label in Name** | A | When avatar labels a control |
| **4.1.2 Name, Role, Value** | A | Named image / group |

---

## 5. Checklist

- [x] Informative avatars expose a person/entity name
- [x] Decorative avatars do not duplicate adjacent text
- [x] Image `alt` uses `name` (no redundant “image of”)
- [x] Initials / icon fallbacks have `role="img"` + accessible name
- [x] Group excess count is text; group labeled if needed
- [x] Status not color-only
- [x] Interactive wrappers own the accessible name and focus
