# Alert — Accessibility Requirements

Guide for implementing and testing accessibility of the Dreamy UI `Alert` component.

**APG pattern:** [Alert](https://www.w3.org/WAI/ARIA/apg/patterns/alert/) (inline / status messages)

---

## 1. Purpose and semantics

`Alert` is an **inline message** (info, success, warning, error). It is **not** a modal dialog and must not use `role="alertdialog"` or focus trapping.

| Use | Role |
| --- | --- |
| Important, time-sensitive interruption (often errors) | `role="alert"` (assertive live) |
| Advisable / ongoing status that should not interrupt | `role="status"` (polite live) |
| Static page content that is present on load | Landmark-free container with clear heading text may suffice; live announcement only when the message **appears dynamically** |

Prefer matching `status` prop intent: e.g. `error` → alert; `info` / `success` → status — unless product requirements differ.

---

## 2. Component anatomy

| Part | Purpose in a11y |
| --- | --- |
| **Root** (`Alert`) | Host for role / live semantics |
| **Title** (`data-part="title"`, rendered as `h4`) | Primary message / accessible name source |
| **Description** (`data-part="description"`) | Supporting detail |
| **StatusIcon** | Decorative (`aria-hidden`); meaning comes from text + status |

---

## 3. Requirements

- Set `role="alert"` or `role="status"` on the root as appropriate; do not rely on color alone (1.4.1).
- Title and description must be readable text; icons are supplementary (`aria-hidden`).
- When the alert is injected dynamically after load, live semantics must cause announcement (4.1.3). Alerts present in the initial HTML may not re-announce.
- Do **not** auto-move focus into the alert unless following a deliberate focus-management pattern for errors.
- Contrast: text ≥ 4.5:1; non-text status affordances ≥ 3:1 (1.4.3 / 1.4.11).

## 4. WCAG (relevant)

| SC | Level | Notes |
| --- | --- | --- |
| **1.4.1 Use of Color** | A | Status + icon + text |
| **1.3.1 Info and Relationships** | A | Message structure programmatic |
| **4.1.2 Name, Role, Value** | A | Correct alert/status role |
| **4.1.3 Status Messages** | AA | Dynamic appearance announced |

---

## 5. Checklist

- [x] Root role is `alert` or `status` (not `alertdialog`)
- [x] Message meaning does not depend on color alone
- [x] Icons are `aria-hidden` (or otherwise non-noisy)
- [x] Dynamic alerts are announced; no unexpected focus move
- [x] Text contrast meets AA
