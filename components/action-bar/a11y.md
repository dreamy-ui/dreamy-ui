# Action Bar — Accessibility Requirements

Guide for implementing and testing accessibility of the Dreamy UI `ActionBar` compound component.

**Toolbar-like surface** for bulk / contextual actions (often shown when items are selected). Prefer the [APG Toolbar Pattern](https://www.w3.org/WAI/ARIA/apg/patterns/toolbar/) for the interactive action cluster; dialog semantics only when the bar truly behaves as a modal dialog.

---

## 1. Purpose and semantics

| Part | Dreamy export | A11y role |
| --- | --- | --- |
| Root | `ActionBar.Root` | Portal + open state; not the named widget |
| Content | `ActionBar.Content` | Interactive surface — prefer `toolbar` (or keep `dialog` only if modal) |
| Selection trigger | `ActionBar.SelectionTrigger` | Status / count text; not a second toolbar |
| Separator | `ActionBar.Separator` | `role="separator"` (already set) |
| Close | `ActionBar.CloseTrigger` | Named button that dismisses |

Current implementation sets `role="dialog"` + `aria-label="Action bar"` on Content. That is acceptable **only** if the bar is modal (focus trap, Esc to close, inert background). For a non-modal floating action strip, prefer:

- `role="toolbar"` on Content (or an inner actions group)
- Accessible name via `aria-label` / `aria-labelledby` (e.g. “Actions” or selection summary)

---

## 2. Naming (required)

| Element | Requirement |
| --- | --- |
| Content | Stable accessible name (`aria-label` or labelled by selection summary) |
| CloseTrigger | Accessible name (default `"Close action bar"` is fine; overridable) |
| Action buttons (consumers) | Each control has its own name; icon-only → `aria-label` |
| SelectionTrigger | Visible text should describe selection (e.g. “3 selected”); do not leave empty |

WCAG [2.4.6 Headings and Labels](https://www.w3.org/WAI/WCAG22/Understanding/headings-and-labels.html), [2.5.3 Label in Name](https://www.w3.org/WAI/WCAG22/Understanding/label-in-name.html), [4.1.2 Name, Role, Value](https://www.w3.org/WAI/WCAG22/Understanding/name-role-value.html).

---

## 3. Keyboard interaction

Align with [APG Toolbar](https://www.w3.org/WAI/ARIA/apg/patterns/toolbar/) when using toolbar semantics:

| Key | Requirement |
| --- | --- |
| **Tab** / **Shift+Tab** | Move focus into / out of the bar (one tab stop into the toolbar is ideal; arrows move within) |
| **Arrow Left/Right** (toolbar) | Move among action controls |
| **Home** / **End** (toolbar) | First / last control |
| **Enter** / **Space** | Activate focused control |
| **Esc** | Dismiss when open (`onClose`); required if treated as dialog |

Also:

- No keyboard trap unless modal dialog with Esc + close ([2.1.2](https://www.w3.org/WAI/WCAG22/Understanding/no-keyboard-trap.html)).
- On open, move focus into the bar (first action or close); on close, restore focus to the invoking control / selection UI ([APG Dialog](https://www.w3.org/WAI/ARIA/apg/patterns/dialog-modal/) if modal).
- Disabled actions are not operable and are announced as disabled.

---

## 4. Open / close and motion

- Open state must be controllable and reflected (`isOpen` / `defaultIsOpen` / `onOpen` / `onClose`).
- Portal content must not be reachable when closed (unmount / inactive portal — already tied to `isOpen`).
- Enter/exit motion must respect reduced motion ([2.3.3](https://www.w3.org/WAI/WCAG22/Understanding/animation-from-interactions.html)).

---

## 5. Separator

- Keep `role="separator"` and `aria-orientation="vertical"` when vertical.
- Decorative-only separators may use `aria-hidden="true"` instead of separator role if they convey no structure.

## 6. WCAG 2.2 (relevant)

| SC | Level | Why |
| --- | --- | --- |
| **1.3.1 Info and Relationships** | A | Toolbar/dialog structure; separator |
| **1.4.11 Non-text Contrast** | AA | Bar chrome, icons, focus |
| **2.1.1 Keyboard** | A | All actions + dismiss |
| **2.1.2 No Keyboard Trap** | A | Esc / Tab out when non-modal |
| **2.4.3 Focus Order** | A | Logical among actions |
| **2.4.7 Focus Visible** | AA | Visible focus on actions |
| **2.4.11 Focus Not Obscured (Minimum)** | AA | Focused control not fully covered |
| **2.5.8 Target Size (Minimum)** | AA | ≥ 24×24 CSS px for actions |
| **4.1.2 Name, Role, Value** | A | Named toolbar/dialog + named buttons |

---

## 7. Checklist

- [ ] Content has an accessible name
- [ ] Role matches behavior: `toolbar` (non-modal) or true modal `dialog`
- [ ] Close control named and dismisses via click + Esc (if dialog/toolbar dismissible)
- [ ] All actions keyboard-operable; focus restored on close
- [ ] Icon-only actions have `aria-label`
- [ ] Selection summary text is meaningful when shown
- [ ] Closed bar content not focusable
- [ ] Reduced motion respected for enter/exit
- [ ] Focus visible; target size OK
