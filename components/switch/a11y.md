# Switch — Accessibility Requirements

Guide for implementing and testing accessibility of the Dreamy UI `Switch` component.

**Implementation pattern:** multipart control built from a root `<label>`, a visually hidden `input[type="checkbox"]`, a visual control track, a thumb, and optional inline text (`children`). Prefer the [APG Switch Example Using HTML Checkbox Input](https://www.w3.org/WAI/ARIA/apg/patterns/switch/examples/switch-checkbox/) as the primary reference for this architecture.

---

## Dreamy form composition (important)

Dreamy form controls (Switch, Checkbox, Input, Select, …) are **standalone controls**. They are meant to be composed **inside `Field`**.

| Concern | Owned by | Not owned by Switch |
| --- | --- | --- |
| Visible field label | `Field.Label` | Do **not** add a built-in field label to Switch |
| Hint / description | `Field.Hint` (or equivalent) | Do **not** bake hint UI into Switch |
| Error message | `Field.Error` | Do **not** bake error UI into Switch |
| Required indicator | `Field` / `Field.Label` | Switch only reflects `required` / `aria-required` on the input |
| `id`, `aria-describedby`, invalid/disabled wiring | `Field` context → Switch input via `useField` | Switch must **consume** Field context, not reimplement Field chrome |

**Implication for fixes and tests:** make Switch correctly **expose hooks for Field** (stable `id`, `aria-labelledby` / label association, `aria-describedby`, `aria-invalid`, disabled/required). Do **not** add Field-like label/hint/error markup into the Switch component itself.

Optional `children` on Switch are an **inline adjacent name** (common for toggles next to short text). That is separate from `Field.Label`. Preferred documented usage:

```tsx
<Field>
  <Field.Label>Email notifications</Field.Label>
  <Switch />
  <Field.Hint>We will only send product updates.</Field.Hint>
  <Field.Error>This setting is required.</Field.Error>
</Field>
```

Accessibility requirements below still say the control **must have** an accessible name, descriptions when invalid, etc. In Dreamy, those usually come **from Field**, not from new UI inside Switch.

---

## 1. Purpose and semantics

A switch is a binary input: **on** or **off**.

- Prefer `switch` semantics when the UI means “on/off” (e.g. notifications, dark mode).
- Prefer checkbox semantics when the UI means “checked/unchecked” in a list of options.
- Switches must **not** expose a third / mixed state. `aria-checked="mixed"` is invalid for `role="switch"` and must be treated as off/`false`.
- The **accessible name must not change** when the switch toggles (no swapping “Enable” ↔ “Disable” as the name).

## 2. Component anatomy (multipart)

| Part | Element (Dreamy) | Role in a11y tree | Purpose |
| --- | --- | --- | --- |
| **Root** | `<label>` | Presentation / click association (not the switch widget) | Wraps parts; activates the input via native association |
| **Input** | visually hidden `<input type="checkbox">` | **The switch widget** | Focus target, keyboard target, form value, name/role/state |
| **Control** | `<span data-part="control">` | Decorative (`aria-hidden="true"`) | Visual track; not announced or focusable |
| **Thumb** | thumb inside control | Decorative | Visual on/off indicator |
| **Inline text** | optional `<span data-part="label">` (`children`) | May contribute to accessible name | Short text beside the track — **not** a replacement for `Field.Label` |

### What must be connected to what

```
┌─ Field ──────────────────────────────────────────────────────────┐
│  Field.Label (id) ──htmlFor / aria-labelledby──► Switch input    │
│  Field.Hint  (id) ──┐                                            │
│  Field.Error (id) ──┴──aria-describedby────────► Switch input    │
│                                                                  │
│  ┌─ Switch root <label> ───────────────────────────────────────┐ │
│  │                                                             │ │
│  │  ┌─ Input (hidden checkbox) ◄── FOCUS + ROLE + STATE ────┐ │ │
│  │  │  id (from Field / props)                              │ │ │
│  │  │  role="switch"                                        │ │ │
│  │  │  checked (native)                                     │ │ │
│  │  │  labelled by Field.Label and/or aria-label/children   │ │ │
│  │  │  aria-describedby ← Field hint/error ids              │ │ │
│  │  └───────────────────────────────────────────────────────┘ │ │
│  │                                                             │ │
│  │  ┌─ Control ── aria-hidden="true" ───────────────────────┐ │ │
│  │  │  └─ Thumb (decorative)                                │ │ │
│  │  └───────────────────────────────────────────────────────┘ │ │
│  │                                                             │ │
│  │  ┌─ Optional children (inline text) ─────────────────────┐ │ │
│  │  │  Adjacent name only; Field.Label remains primary      │ │ │
│  │  └───────────────────────────────────────────────────────┘ │ │
│  └─────────────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────────────┘
```

**Rules:**

1. **Only the input** is the interactive switch in the accessibility tree.
2. Root associates clicks with the input; it must not become a second widget (`role="switch"` on root while input also owns the role).
3. Control + thumb are **presentational** (`aria-hidden="true"` on the control).
4. Accessible name comes from **Field.Label** (preferred), and/or `aria-label`, and/or optional inline `children` — without inventing Field chrome inside Switch.
5. Hint/error descriptions connect to the **input** via `aria-describedby` from Field context.

## 3. Per-part requirements

### 3.1 Input (the switch)

**Role**

- Set `role="switch"` on the `input[type="checkbox"]` (APG checkbox-based example).
- That remaps AT announcement from “checkbox” to “switch” (on/off).

**State**

- Expose on/off via the native **`checked`** property/attribute.
- Per APG: when the switch element **is** an `input[type="checkbox"]`, use HTML `checked` **instead of** `aria-checked`.
- Do **not** use `aria-checked="mixed"` for switch semantics.

**Name (required — usually via Field)**

The control must have an accessible name. In Dreamy, provide it by composing Field (do not add a Field label into Switch):

| Method | When |
| --- | --- |
| `Field.Label` ↔ input (`htmlFor` / `id` / Field `getLabelProps`) | **Preferred** for form usage |
| `aria-label` on Switch / input | When there is no visible Field label (icon-only, compact UI) |
| Optional `children` inline text | Extra adjacent text; still prefer Field.Label for the primary form label |

WCAG 2.5.3: when a visible label exists (`Field.Label` or children), the accessible name should contain that visible text.

**Other states / properties on the input**

| Attribute | Requirement |
| --- | --- |
| `disabled` | Prefer native `disabled` from Field / props |
| `aria-invalid` | From Field `isInvalid` |
| `aria-describedby` | Field hint + error element ids |
| `required` / `aria-required` | From Field `isRequired` |
| `name` / `value` | Form submission |
| `id` | Shared with Field for label association |

**Focus**

- Input is the tab stop.
- Visual focus may paint on the control (`data-focus-visible`); DOM focus stays on the input.

### 3.2 Root (`<label>`)

- Click/activation association with the input.
- Not the focusable switch widget; no stealing focus from the input.
- No conflicting `role="switch"` / `aria-checked` on root if the input already owns those semantics.
- May use `data-*` for CSS state (`data-checked`, `data-disabled`, `data-invalid`, …).

### 3.3 Control (track)

- `aria-hidden="true"`.
- Not focusable; no alternate interaction path that bypasses the input.
- Visual state stays in sync with input `checked`.

### 3.4 Thumb

- Decorative only; no role/name/tabindex.
- Icons inside thumb should not create a conflicting accessible name (`aria-hidden` if needed).

### 3.5 Inline text (`children`)

- Optional short text beside the track.
- Stable wording across on/off.
- **Not** a substitute for implementing Field; primary form labeling belongs on `Field.Label`.
- If APG-style visual “On”/“Off” state words are ever shown, hide them from AT with `aria-hidden="true"`.

## 4. Keyboard interaction

From the [Switch Pattern](https://www.w3.org/WAI/ARIA/apg/patterns/switch/) and checkbox-based example:

| Key | Requirement |
| --- | --- |
| **Tab** / **Shift+Tab** | Moves focus to / from the switch (the input) |
| **Space** | Toggles on ↔ off when focused |
| **Enter** | Optional; APG allows Enter to toggle. If implemented, keep behavior consistent |

Also:

- No keyboard trap (WCAG 2.1.2).
- Disabled / read-only must not toggle.
- Pointer activation on root/control must toggle the same input as keyboard.

---

## 5. Grouping multiple switches

When several switches share a visible group label (outside a single Field):

- Wrap in `<fieldset>` + `<legend>`, **or**
- `role="group"` + `aria-labelledby` → group label id.

Each switch still has its own name (typically each in its own `Field`, or one shared Fieldset with multiple Fields).

---

## 6. Field integration (required composition model)

Switch must work correctly **inside Field**. Field owns chrome; Switch owns the widget.

| Concern | Field provides | Switch must |
| --- | --- | --- |
| Label | `Field.Label` | Accept Field `id` / label props so the input is named |
| Hint | `Field.Hint` | Merge hint id into input `aria-describedby` |
| Error | `Field.Error` | Merge error id into `aria-describedby`; set `aria-invalid` |
| Required | `isRequired` | Reflect on input |
| Disabled / read-only | Field flags | Reflect on input + visual `data-*` |

**Do not** fix a11y by shipping built-in Field.Label / Hint / Error inside `Switch`.

## 7. WCAG 2.2 success criteria (relevant)

Mapped from the [WCAG 2.2 Quickref](https://www.w3.org/WAI/WCAG22/quickref/).

### Perceivable

| SC | Level | Why it applies to Switch |
| --- | --- | --- |
| **1.1.1 Non-text Content** | A | Track/thumb/icon get their alternative via the control’s accessible name |
| **1.3.1 Info and Relationships** | A | Field label / hint / error relationships must be programmatic |
| **1.3.3 Sensory Characteristics** | A | Instructions must not rely only on color/position |
| **1.4.1 Use of Color** | A | On/off not by color alone (thumb position, etc.) |
| **1.4.3 Contrast (Minimum)** | AA | Field label / inline text contrast |
| **1.4.11 Non-text Contrast** | AA | Track / thumb / focus ring ≥ 3:1 |
| **1.4.13 Content on Hover or Focus** | AA | If overlays appear on hover/focus, meet dismiss/hover/persist rules |

| SC | Level | Why it applies to Switch |
| --- | --- | --- |

| **2.1.2 No Keyboard Trap** | A | Focus can leave |

| **2.4.7 Focus Visible** | AA | Visible focus indicator |

| **2.4.13 Focus Appearance** | AAA | Optional stronger focus target |

| **2.5.3 Label in Name** | A | Name contains visible Field label / children text |
| **2.5.8 Target Size (Minimum)** | AA | ≥ 24×24 CSS px (or spacing exception) |

| SC | Level | Why it applies to Switch |
| --- | --- | --- |
| **3.2.1 On Focus** | A | Focus must not unexpectedly change context |

| **3.3.2 Labels or Instructions** | A | Via `Field.Label` / `Field.Hint` |

## 8. Visual / motion requirements (APG accessibility features)

- On/off distinguishable beyond color (thumb position).
- Focus/hover should clarify the interactive surface.
- Pointer cursor on interactive surface.
- Avoid transparent-only focus borders (forced-colors / high contrast).
- Support `prefers-reduced-motion` / Dreamy `reduceMotion` for thumb animation (WCAG 2.3.3).

---

## 9. Checklist (implementation / tests)

- [ ] Focus lands on the **hidden checkbox input**
- [ ] Input has `role="switch"`
- [ ] Input exposes on/off via native `checked`
- [ ] Works inside **Field**: label, hint, error wire to the input (no Field chrome added into Switch)
- [ ] Accessible name present when used with `Field.Label` or `aria-label` / children
- [ ] Accessible name does **not** change when toggled
- [ ] Space toggles; Tab moves focus
- [ ] Control/thumb are `aria-hidden` (or excluded from the name)
- [ ] Disabled: not operable; announced as disabled
- [ ] Invalid + hint/error: `aria-invalid` + `aria-describedby` when used with Field
- [ ] No second conflicting `role="switch"` / `aria-checked` on the root
- [ ] Visible Field label text included in accessible name (2.5.3)
- [ ] Focus visible; non-text contrast OK (1.4.11)
- [ ] Reduced motion respected for thumb animation
