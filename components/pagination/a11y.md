# Pagination — Accessibility Requirements

Guide for implementing and testing accessibility of the Dreamy UI `Pagination` component.

**Related APG:** navigation landmarks and link/button naming; no dedicated pagination pattern — apply [APG practices](https://www.w3.org/WAI/ARIA/apg/patterns/) for buttons and landmarks.

---

## 1. Purpose and semantics

Pagination lets users move between pages of results. Controls must announce **which page is current** and remain fully operable by keyboard.

---

## 2. Component anatomy

| Part | Element (Dreamy) | Role in a11y tree | Purpose |
| --- | --- | --- | --- |
| **Root** | `<nav>` | Navigation landmark | Groups pagination controls |
| **Item** | `IconButton` | Button | Jump to a page; `aria-current="page"` when selected |
| **Ellipsis** | `<span aria-hidden>` | Hidden | Visual gap between page ranges |
| **PrevTrigger** / **NextTrigger** | `IconButton` | Button | Adjacent page; named via `aria-label` |
| **PageText** | `<span>` | Text | Optional “1 of 10” / range summary |
| **Items** | renders Item + Ellipsis | — | Range helper |

```
nav[aria-label] → Prev | Item[aria-current=page]… | Ellipsis[aria-hidden] | Next | optional PageText
```

---

## 3. Requirements

### Landmark and name

- Root is `<nav>` with a distinctive label, e.g. `aria-label="Pagination"` (or more specific: “Search results pages”).

### Current page announcement

- Selected page control sets `aria-current="page"` (Dreamy `usePaginationItem`).
- Page buttons expose a clear name: `aria-label="Page N"` (or visible text that is uniquely understood).
- Do **not** rely on color alone to mark the current page (WCAG 1.4.1); keep `aria-current` in sync with visuals.

### Prev / next

- Icon-only controls need accessible names (`aria-label="Previous page"` / `"Next page"`).
- Decorative chevron icons: `aria-hidden="true"`.
- Disable (and expose disabled) when at first/last page.

### Ellipsis

- Presentational only (`aria-hidden="true"`); not focusable.

### Keyboard

| Key | Requirement |
| --- | --- |
| **Tab** / **Shift+Tab** | Moves among prev, page buttons, next |
| **Enter** / **Space** | Activates focused button |

- No keyboard trap.
- Changing page must not trap focus; keep focus on a logical control or move focus to updated content if the product pattern requires it.
- Optional: announce page change via live region / `PageText` update when content reloads asynchronously (WCAG 4.1.3).

### PageText

- Visible summary (“1 of 10”) helps sighted and AT users; keep it in sync with `page` / `totalPages`.
- Prefer associating it with the nav via proximity or `aria-describedby` when useful.

| SC | Level | Why |
| --- | --- | --- |
| **1.4.1 Use of Color** | A | Current page not by color alone |
| **2.1.1 Keyboard** | A | All page controls operable |
| **2.4.3 Focus Order** | A | Logical control order |
| **2.4.7 Focus Visible** | AA | Button focus |
| **2.5.8 Target Size (Minimum)** | AA | ≥ 24×24 CSS px |
| **4.1.2 Name, Role, Value** | A | Named buttons; `aria-current` |

---

## 5. Checklist

- [ ] Named `<nav>` landmark
- [ ] Current page has `aria-current="page"`
- [ ] Each page control has an accessible name
- [ ] Prev/next have `aria-label`; icons `aria-hidden`
- [ ] Ellipsis hidden from AT
- [ ] Disabled prev/next at ends
- [ ] Full keyboard operation; focus visible
- [ ] Current page distinguishable beyond color
