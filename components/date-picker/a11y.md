# DatePicker — Accessibility Requirements

Guide for implementing and testing accessibility of the Dreamy UI `DatePicker` component.

**Primary APG references:** [Dialog (Modal)](https://www.w3.org/WAI/ARIA/apg/patterns/dialog-modal/) / popover focus management; [Date Picker Dialog Example](https://www.w3.org/WAI/ARIA/apg/patterns/dialog-modal/examples/datepicker-dialog/); grid keyboard model similar to [Grid Pattern](https://www.w3.org/WAI/ARIA/apg/patterns/grid/).

---

## Dreamy form composition

DatePicker is a **standalone control**. Compose the trigger/input **inside `Field`**. Do **not** add Field label/hint/error chrome into DatePicker.

```tsx
<Field>
  <Field.Label>Start date</Field.Label>
  <DatePicker.Root>
    <DatePicker.Input />
    <DatePicker.Content>
      <DatePicker.Header />
      <DatePicker.Calendar />
    </DatePicker.Content>
  </DatePicker.Root>
  <Field.Error>Enter a valid date.</Field.Error>
</Field>
```

---

## 1. Purpose and semantics

Users pick a **single date** via a read-only (or editable Control) field that opens a popover calendar.

- Trigger/Input opens the popover; calendar is a focusable grid of day (or month/year) buttons.
- Selected value must be programmatically exposed on the field (visible text + form-associated value where applicable).
- Min/max dates disable out-of-range cells.

---

## 2. Component anatomy (multipart)

| Part | Element (Dreamy) | Role in a11y tree | Purpose |
| --- | --- | --- | --- |
| **Root** | Popover + context | Owns open state | Wraps trigger + content |
| **Input** / **Trigger** | read-only input group or button | Opens dialog | Named by Field; shows formatted date |
| **Control** | `input[type="date"]` + Today button | Native date field | Optional editable path |
| **Content** | popover/dialog content | Overlay | Focus moves into calendar on open |
| **Header** / nav buttons | icon buttons | Buttons | Prev/next month or year page (`aria-label`) |
| **Nav** | Day/Month/Year buttons | Buttons | Switch calendar view |
| **Calendar** grid | `role="grid"` (day view) | Grid | Arrow-key navigation among cells |
| **Cell buttons** | `<button>` | Gridcell buttons | `aria-label` with full date; roving `tabIndex` |
| **Footer** | Cancel / Apply | Buttons | Confirm or dismiss when present |

---

## 3. Per-part requirements

### Input / Trigger

- Accessible name from **Field.Label** (or `aria-label`).
- Convey current value in the accessible name/value (formatted text in the input).
- `aria-expanded` / `aria-haspopup` (or Popover equivalents) should reflect open state when exposed by the popover trigger.
- Calendar icon decorative (`aria-hidden`) unless it is the sole control.

### Calendar grid (day view)

- Container: `role="grid"` with weekday headers.
- Only **one** day button in tab order (`tabIndex={0}`); others `-1` (roving tabindex).
- Each day button: clear `aria-label` (e.g. “Monday, January 5, 2026”); selected/today/disabled states reflected.
- Disabled dates not operable.

### Header nav buttons

- Icon-only: required `aria-label` (“Previous month”, “Next month”, etc.).
- Visible month/year title should be available to AT (heading or labelled region).

### Month / year views

- Same keyboard model: roving tabindex, Enter/Space select, arrows move focus.

### Focus management

- On open: move focus into the calendar (focused/selected/today cell).
- On close: restore focus to the trigger/input.
- Escape closes the popover; no keyboard trap (2.1.2).

---

## 4. Keyboard interaction

| Key | Context | Requirement |
| --- | --- | --- |
| **Enter** / **Space** | Trigger | Open popover |
| **Escape** | Open | Close; return focus to trigger |
| **Arrows** | Day/month/year grid | Move focus among cells; change month/page when crossing edges |
| **Enter** / **Space** | Focused cell | Select date (or month/year) |
| **Tab** | Open | Cycle focusable controls in the popover (nav, views, footer) |

---

## 5. WCAG highlights

| SC | Why |
| --- | --- |
| **1.3.1 / 4.1.2** | Name, expanded state, selected date |
| **2.1.1 / 2.4.3 / 2.4.7** | Full keyboard use; logical focus; visible focus |
| **2.4.11** | Focused cell not fully obscured |
| **2.5.8** | Day cells / icon buttons ≥ 24×24 CSS px |
| **3.3.1 / 3.3.2** | Labels and errors via Field |
| **3.2.2** | Selecting a date should not surprise-navigate away |

---

## 6. Checklist

- [x] Used inside **Field** for label / hint / error
- [x] Trigger/input has accessible name and shows current value
- [x] Open: focus → calendar; Close: focus → trigger
- [x] Day grid: `role="grid"`, roving tabindex, full-date `aria-label`s
- [x] Prev/next icon buttons labelled
- [x] Min/max disabled correctly
- [x] Escape closes; no trap
- [x] Reduced motion respected for layout animations
