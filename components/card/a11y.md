# Card — Accessibility Requirements

Guide for implementing and testing accessibility of the Dreamy UI `Card` compound component.

**Anatomy:** `Root` (`div`) → `Header` / `Body` / `Footer` → optional `Title` (`h3`) + `Description` (`p`).

---

## 1. Purpose and semantics

Card is a **layout / grouping** pattern. By default it is not a widget and needs no ARIA role.

---

## 2. Structure

| Part | Element | Requirement |
| --- | --- | --- |
| **Root** | `div` | Group content visually; no `role="button"` / `role="link"` by default |
| **Title** | `h3` (default) | Semantic heading; change level to fit the page outline |
| **Description** | `p` | Supporting text |
| **Header / Body / Footer** | `div` | Presentational regions; use landmarks only when justified |

### Interactive cards

Cards often wrap a link or action. Prefer one of these patterns:

1. **Nested control (preferred):** keep Root presentational; put a real `<a>` / `<button>` (e.g. on the title or a CTA) with its own name, focus, and keyboard behavior.
2. **Whole-card control:** if the entire card is the hit target, the interactive element must have:
   - correct role (`link` or `button` — not both)
   - keyboard support (Enter/Space as appropriate)
   - visible focus
   - accessible name that includes visible title text (WCAG 2.5.3)
   - no nested interactive elements (avoid link-inside-link)

Do **not** attach `onClick` to a non-interactive `div` without role, keyboard handlers, and focus styles.

---

## 4. Checklist

- [x] Static cards have no fake widget role
- [x] Title is a real heading at the correct level
- [x] Interactive behavior uses a real control (preferred) or a fully keyboard-accessible whole-card pattern
- [x] No nested interactive conflicts
- [x] Focus visible on any interactive card control
