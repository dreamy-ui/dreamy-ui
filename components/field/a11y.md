# Field — Accessibility Requirements

Guide for implementing and testing accessibility of the Dreamy UI `Field` component.

**Field owns form chrome.** Other Dreamy controls (Input, Switch, Slider, FileUpload, DatePicker, …) are widgets that **consume** Field context. They must **not** reimplement label / hint / error UI.

**Primary references:** [WCAG 3.3.2 Labels or Instructions](https://www.w3.org/WAI/WCAG22/Understanding/labels-or-instructions.html); native `<label>` association; [ARIA `aria-describedby` / `aria-invalid`](https://www.w3.org/TR/wai-aria-1.2/).

---

## 1. Purpose and semantics

Field wraps a single control (or a tightly coupled control group) and provides:

- Accessible **name** via `Field.Label`
- **Instructions** via `Field.Hint`
- **Error identification** via `Field.Error`
- Shared **id** + describedby wiring for the child control through `useField` / Field context

```tsx
<Field isRequired isInvalid={!!error} error={error} hint="Use your work email.">
  <Field.Label>Email</Field.Label>
  <Input />
  {/* or shortcut: <Field label="Email" hint="…" error="…"><Input /></Field> */}
</Field>
```

---

## 2. Component anatomy (multipart)

| Part | Element (Dreamy) | Role in a11y tree | Purpose |
| --- | --- | --- | --- |
| **Root** | `<div>` + Field provider | Grouping container | Supplies context (`id`, invalid, required, disabled, …) |
| **Label** | `<label>` | Label | Names the control (`htmlFor` ↔ control `id`) |
| **RequiredIndicator** | `<span>` | Part of label | Visual required cue; must not be the only cue (`required` / `aria-required` on control) |
| **Hint** | description element | Description | Linked via `aria-describedby` when not invalid |
| **Error** | error message element | Description / alert content | Linked via `aria-describedby` when invalid; visible text required |
| **ErrorIcon** | decorative icon | Hidden / supplemental | Must not replace error text; keep decorative or redundant |

---

## 3. What Field must provide to controls

| Concern | Field responsibility |
| --- | --- |
| Stable `id` | Control uses same `id`; Label uses matching `htmlFor` |
| Label association | `getLabelProps` → native label binding |
| Descriptions | Hint and Error ids merged into control `aria-describedby` |
| Invalid | `isInvalid` → control `aria-invalid` |
| Required | `isRequired` → control `required` / `aria-required` + indicator |
| Disabled / read-only | Flags available to child via context / `get*Props` |

**Mutual exclusivity:** when invalid, prefer showing **Error** and hiding **Hint** (current Dreamy behavior) so `aria-describedby` points at the actionable message.

---

## 4. Per-part requirements

### Label

- Visible text preferred (WCAG 2.5.3 Label in Name).
- Associate with the control via `htmlFor` / `id` (or `aria-labelledby` if the pattern requires it).
- Required indicator: visible “*” or text; ensure the control also exposes required state programmatically.

### Hint

- Helpful instructions before error state (3.3.2).
- Must have an `id` referenced by the control’s `aria-describedby`.

### Error

- Render only when invalid; always include **text** (not color alone — 1.4.1 / 3.3.1).
- Prefer suggestions when possible (3.3.3).
- `id` referenced from the control’s `aria-describedby`.
- Optional `role="alert"` / live region only when appropriate for dynamic errors (4.1.3); avoid noisy announcements on every keystroke unless intentional.

### ErrorIcon

- Decorative relative to Error text (`aria-hidden`), or redundant text alternative — never the sole error cue.

---

## 5. Composition rules (critical)

1. **One Field → one primary labeled control** (or one composite whose value is labeled as a unit).
2. Child controls **consume** Field; they do not ship their own Field.Label / Hint / Error.
3. Nested interactive chrome inside the control (e.g. Switch inline children) is **not** a substitute for `Field.Label`.
4. For multiple related controls, use **Fieldset** + multiple Fields (or Fieldset legend + Fields).

---

## 6. WCAG highlights

| SC | Why |
| --- | --- |
| **1.3.1** | Label/hint/error relationships programmatic |
| **2.4.6 / 3.3.2** | Clear labels and instructions |
| **2.5.3** | Accessible name contains visible label text |
| **3.3.1 / 3.3.3** | Error text (+ suggestions when possible) |
| **4.1.2** | Name/description wiring to the control |

---

## 7. Checklist

- [x] Label associated with control `id`
- [x] Hint and/or Error ids appear in control `aria-describedby` as appropriate
- [x] Invalid → `aria-invalid` on control + visible Error text
- [x] Required → indicator + programmatic required on control
- [x] No reliance on color-only for errors
- [x] Child control does not duplicate Field chrome
